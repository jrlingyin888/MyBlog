> 2026 年 5 月 14 日，Anthropic 在 Claude 官方博客发布了《The Founder's Playbook: Building an AI-Native Startup》（AI 原生创业手册，约 35 页）。它第一次按照 2026 年的 AI 能力，把经典的创业四阶段——**Idea / MVP / Launch / Scale**——重新画了一遍，并为每个阶段定义了「目标、进入下一阶段的退出标准、常见的失败模式」。
>
> 这篇文章把整本手册按章节拆开，并补上一件原文没替你做完的事：**每个阶段，Claude Chat / Claude Code / Claude Cowork 分别该干什么、你具体该怎么问。** 拿来即用。

---

## 一、先理解它的内核：一句话

整本手册的前提只有一句：

> **AI 降低的是「执行」的门槛，而不是「判断力」的门槛。**

过去卡住创业每个环节的三样东西——**资本、人手、技术技能**——被 AI 抹掉了。没写过代码的人也能发布生产级软件，两三个人的团队能跑出过去几十人的覆盖面。

于是创始人的角色发生了根本转变：

- **过去**：你要么自己写代码，要么招工程师，亲自做执行。
- **现在**：你是一个**指挥家 / 架构师**，编排一队 AI agents 去执行，把自己最稀缺的注意力留给「只有创始人能做的事」——客户对话、定位决策、文化建设。

Anthropic 自曝了一个内部数据：他们把「从想法到上线」的周期从 **6 个月压缩到了一天**，内部口头禅是 **"Just do things"**。当试错成本从「浪费几个月」掉到「一个下午做个原型」，最优策略自然从「想清楚再做」变成了「先做出来再说」。

**这对你意味着什么**：你要练的核心技能不再是「能不能造出来」，而是「能不能比别人更快地迭代和编排」。

## 二、四阶段逐章拆解

每个阶段我用同一套结构讲清楚：**这阶段的目标 → 退出标准（达到才进下一阶段）→ 最容易死的坑 → 三个 Claude 工具怎么分工 → 你具体怎么问。**

---

### 阶段 1 · Idea（想法）—— 目标不是「想出点子」，而是「验证一个值得解决的问题」

Idea 阶段的目标常被误解。手册强调：你要做的不是憋一个绝妙创意，而是**验证一个真实存在、有人愿意付费解决的问题**。它要求你用 AI 做三件事：**问题验证、竞品扫描、客户洞察**。

| 关键动作 | 退出标准（达到才算过关） | 典型失败模式 |
|---|---|---|
| 问题验证 | 找到 10 个愿意付费的目标用户 | 沉迷于解决方案，忽略问题本身 |
| 竞品扫描 | 能清晰说出与现有方案的 3 个关键差异 | 把「没有竞品」当成卖点 |
| 客户洞察 | 记录至少 20 段一手、原始的用户对话 | 用问卷代替深度访谈 |

**三个 Claude 工具的分工：**

- **Claude Chat（主力）**：当你的研究员和军师。让它读几百份用户访谈记录、自动总结竞品定位差异、根据公开信号生成「目标客户访谈名单」。
- **Claude Code**：这阶段几乎不用，最多让它快速搭一个用于收集需求的落地页或问卷工具。
- **Claude Cowork**：暂不需要。

**你具体可以这样问 Claude Chat：**

```
我想做一个 [一句话描述产品]，目标用户是 [谁]。
请帮我：
1. 列出这个问题在 Reddit / 小红书 / 知乎上的高频抱怨，并归纳 5 个最痛的痛点；
2. 找出 3 个现有的竞品/替代方案，分别说明它们的定位和明显短板；
3. 帮我起草一份 10 个开放式问题的用户访谈提纲，目标是验证「这个问题是否真实、是否值得付费」。
```

把访谈做完后，再喂回去：

```
这是我做的 8 段用户访谈记录 [粘贴]。
请帮我：判断哪些是真痛点、哪些只是「有了更好」；
有没有人明确表达了付费意愿？把原话摘出来。
```

> **Idea 阶段的纪律**：不要急着写代码。你的退出标准是「拿到 10 个愿意付钱的人」，不是「做出了一个 demo」。

---

### 阶段 2 · MVP —— 核心是「在 AI 加速下，仍然保持工程纪律」

MVP 阶段最大的反直觉点：**AI 写代码很快，但如果你不约束范围，技术债会以前所未有的速度堆积。** 这是本阶段唯一要时刻提防的事。

- **主力工具**：Claude Code，搭配一个 `CLAUDE.md` 项目记忆文件来维持上下文一致性。
- **多 Agent 团队模式（Multi-Agent Team）**：把 UI、后端、QA 拆给不同的 agent 并行推进。

| 退出标准 | 典型失败模式 |
|---|---|
| 一个**可演示的核心闭环**，且通过最小安全清单（鉴权、API key 管理、依赖审计） | 「Demoware 陷阱」——demo 看着惊艳，但底层数据模型撑不住第二个真实客户的数据冲击 |

**三个 Claude 工具的分工：**

- **Claude Code（绝对主力）**：写代码、跑多 agent 并行、维护 `CLAUDE.md`。
- **Claude Chat**：当你卡在架构选型、安全清单、技术权衡时的咨询对象。
- **Claude Cowork**：仍不需要。

**你具体可以这样用 Claude Code：**

第一步，先让它建立项目记忆（`CLAUDE.md`）：

```
请帮我生成一个 CLAUDE.md 文件，记录：
- 项目目标与核心闭环（一句话）
- 技术栈与目录结构约定
- 编码规范、命名规范
- 「不要做」的清单（防止范围蔓延）
```

第二步，用多 agent 拆活：

```
我要做的核心闭环是 [描述]。请按这个方式推进：
- 用一个 agent 负责 UI 层
- 一个 agent 负责后端 / 数据模型
- 一个 agent 负责写测试和过一遍安全清单（鉴权、密钥管理、依赖审计）
先给我整体方案，我确认后再分别动手。
```

第三步，每完成一块就让它自检：

```
请审查刚才这段代码：
1. 数据模型能不能撑住第二个、第十个真实客户？
2. 有没有把密钥写死、有没有未鉴权的接口？
3. 哪些是为了 demo 走的捷径、上线前必须还的技术债？列个清单。
```

> **MVP 阶段的纪律**：你的退出标准是「一个能撑住真实数据的核心闭环」，不是「一个能演示的漂亮壳子」。

---

### 阶段 3 · Launch（发布）—— 目标是「分辨真实牵引力 vs. 早期虚假繁荣」

Launch 阶段最容易掉进「虚假繁荣」的坑：朋友、同行、早期尝鲜者的热情，会让你误以为产品成了。手册给了三个**客观指标**来区分真假：

1. **留存曲线是否在变平**（而不是持续往下掉）；
2. **用户的主动回访率**；
3. **付费转化的边际成本**（拉新越来越贵还是越来越便宜）。

这一阶段，你要开始用 **Claude Cowork 自动化内部运营**，把自己从杂事里捞出来，专注做销售和融资。手册把这套叫做「**发布操作系统（Launch OS）**」——用多 agent 协作完成 80% 的重复工作（客服、内容生产、社群管理），让创始人专注在只有自己能做的事上。

**三个 Claude 工具的分工：**

- **Claude Cowork（新主力）**：自动化客服回复、内容批量生产、社群日常管理、数据周报。
- **Claude Chat**：分析留存数据、打磨定位话术、准备融资材料 / BP。
- **Claude Code**：持续迭代产品，修 bug、上小功能。

**你具体可以这样问：**

让 Claude Chat 帮你判断牵引力真假：

```
这是我上线后 4 周的数据 [粘贴：日活、留存、付费转化、获客成本]。
请帮我判断：这是真实牵引力还是早期虚假繁荣？
重点看：留存曲线是否变平、回访率、获客边际成本的趋势。
```

让 Claude Cowork 搭「发布操作系统」：

```
帮我把这些重复工作流程化：
- 把常见客服问题整理成 SOP 并自动起草回复；
- 根据产品更新，每周批量生成 3 条社群/社媒内容；
- 每周一自动汇总上周关键指标，生成一页周报。
```

> **Launch 阶段的纪律**：退出标准是「留存曲线变平 + 可重复的付费转化」，不是「一时的下载高峰」。

---

### 阶段 4 · Scale（扩张）—— 关键不是融资，而是搭一套「可复制的 agentic operating system」

Scale 阶段，很多人以为关键是融钱。手册的判断相反：**关键是搭一套可复制的「agent 化运营系统」。** 同时要开始管理一个新东西——**agent 化的技术债**，以及构建**持久化上下文（persistent context）**，让 agent 们跨任务记得住事。

手册给了一套完整的 **Claude 产品矩阵部署建议**，每个产品对应一类「不能外包给人」的高频任务：

| Claude 产品 | 在 Scale 阶段的角色 |
|---|---|
| **Claude apps（Chat）** | 客户支持入口 |
| **Claude Cowork** | 内部知识管理、团队协作中枢 |
| **Claude Code** | 持续的产品代码迭代 |
| **Claude Platform**（API） | 后端模型调用、多 agent 编排 |

**你具体可以这样做：**

让 Claude Chat 帮你设计这套系统：

```
我的产品已经有了稳定付费用户。请帮我设计一套可复制的「agent 运营系统」：
- 哪些高频任务应该交给固定的 agent，而不是人；
- 怎么管理「agent 技术债」，避免越跑越乱；
- 怎么构建持久化上下文，让 agent 跨任务记得住客户和项目状态。
```

> **Scale 阶段的纪律**：你不是在「加人」，而是在「加系统」。传统「团队规模 → 估值倍数」的公式正在失效。

---

## 三、一张表看懂：Claude 产品矩阵该在哪个阶段引入

手册一个被低估的亮点是：它精确地告诉你**每个产品该在哪个阶段进场**——你不用一开始就为「全家桶」买单，而是随业务演进逐步引入。

| Claude 产品 | 主要阶段 | 角色定位 |
|---|---|---|
| **Claude apps（Chat）** | Idea / Launch | 创始人的军师、客户支持 |
| **Claude Code** | MVP / Scale | 工程主力、多 Agent 编排 |
| **Claude Cowork** | Launch / Scale | 团队协作、知识库、运营自动化 |
| **Claude Platform** | Scale | 后端 API、自定义 agent |

## 四、手册的「隐藏彩蛋」：9 个被让出来的消费级 AI 赛道

这本手册之所以引发热议，是因为它和 Anthropic 4 月 30 日发布的《百万对话分析报告》形成了组合拳。报告显示：约 **6% 的 Claude 对话属于「个人建议」**，集中在 9 个高需求的消费级 AI 领域。Anthropic 在手册里明说：**这些 to C 市场它自己不做，留给创业者。**

潜台词很清楚：用户之所以转向 AI，是因为他们「**请不起、找不到、约不上**」专业人士——这本质上是供给侧失灵造就的市场机会。

| 领域 | 典型痛点 | 商业化方向 |
|---|---|---|
| 健康 / 医疗 | 问诊周期长、健康自我管理 | AI 健康管家、慢病数据助手 |
| 职业发展 | 晋升瓶颈、转行规划 | 职业教练、面试模拟 |
| 关系 | 沟通障碍、情绪梳理 | 私人心理陪伴 |
| 金钱 / 理财 | 个人理财、报税 | 私人 CFO 助手 |
| 育儿 | 育儿决策、成长追踪 | 家庭育儿副驾 |
| 法律维权 | 维权门槛高、合同解读 | 法律自助平台 |
| 生命科学 | 学习生物医学知识 | 研究工具与教育 |

最经典的样板是 **Cal AI**：只切「卡路里计算 + 身体成分」这一个垂直点，做到了 4000 万美元营收、5000 万 ARR，**只有 7 个人、0 风投**。这正是 AI 原生时代反复被引用的「超精益创业」模型——小团队深耕单一领域，照样能跑出独立估值。

## 五、最后：手册没说出口的一个提醒

有分析指出，手册推荐用 Claude Cowork 处理「合规相关」的工作流，但 **Cowork 的活动被排除在 Anthropic 的审计日志、Compliance API 和数据导出之外**。如果你的业务涉及合规、审计场景，这一点要留个心眼，别把需要留痕的流程全压在 Cowork 上。

---

## 写在最后：现在就能做的三件事

1. **自我定位**：你现在到底在 Idea / MVP / Launch / Scale 的哪个阶段？对照上面每个阶段的「退出标准」诚实打分。
2. **练编排**：从今天起，把「用一个 Claude 当指挥、拆活给多个 agent」当成日常肌肉来练。
3. **找缝隙**：在 9 个消费级赛道里，按「政策风险 / 数据获取难度 / 付费意愿」给每个赛道打分，挑一个最适合你的切入。

AI 原生创业的胜负手，已经从「你能不能造出来」，变成了「**你能不能比别人迭代和编排得更快**」——这恰恰是这本手册想让每个创始人从今天就开始练的功夫。
